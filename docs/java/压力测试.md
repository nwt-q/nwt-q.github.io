##   压力测试



### 1. 响应时间（Response Time, RT）

**定义**：用户从客户端发起一个请求开始，到客户端接收到服务器端返回的响应结束，整个过程所耗费的总时间。

**核心意义**：直接反映用户体验，是衡量系统 “快慢” 最直观的指标，**数值越小，用户体验越好**。

------

### 2. 吞吐量类核心指标（HPS / TPS / QPS）

这三个指标都是衡量系统**每秒处理能力**的核心吞吐量指标，**数值越大，系统处理能力越强**，三者的区别和适用场景如下：

|  指标   |          全称          |                定义                 |                           适用场景                           |
| :-----: | :--------------------: | :---------------------------------: | :----------------------------------------------------------: |
| **HPS** |    Hits Per Second     |     每秒点击次数，单位：次 / 秒     |     衡量对服务器的单击请求数，多用于前端 / 页面访问场景      |
| **TPS** | Transaction per Second |  系统每秒处理交易数，单位：笔 / 秒  | 衡量**完整业务流程**的处理能力（如一次下单、一次转账，包含多步接口调用） |
| **QPS** |    Query per Second    | 系统每秒处理查询次数，单位：次 / 秒 | 衡量**单个接口 / 查询**的处理能力，多用于接口服务、数据库查询场景 |

#### 关键补充说明

- 特殊等价场景：对于互联网业务中**仅包含一个请求连接的简单业务**，`TPS = QPS = HPS`，三者数值一致。

- 选型原则：

  - 评估**完整业务链路**性能 → 用 **TPS**
  - 评估**单个接口 / 查询**性能 → 用 **QPS**
  - 评估**前端页面点击 / 访问**性能 → 用 **HPS**

  

------

### 3. 不同行业 TPS 参考基准（非互联网化活动场景）



|    行业类型    |     TPS 参考范围      |                  说明                  |
| :------------: | :-------------------: | :------------------------------------: |
|    金融行业    |  1000TPS ~ 50000TPS   | 传统金融核心交易，不包含互联网营销活动 |
|    保险行业    |  100TPS ~ 100000TPS   |    传统保险业务，不包含互联网化活动    |
|    制造行业    |    10TPS ~ 5000TPS    |      工业系统、ERP 等内部业务系统      |
| 互联网电子商务 | 10000TPS ~ 1000000TPS |     电商平台、大促场景，高并发需求     |
| 互联网中型网站 |  1000TPS ~ 50000TPS   |          常规中流量互联网平台          |
| 互联网小型网站 |   500TPS ~ 10000TPS   |       低流量个人 / 小型企业网站        |

------

### 4. 补充指标：最大响应时间（Max Response Time）

**定义**：用户发出请求 / 指令后，系统做出响应的最大耗时。

**核心意义**：用于排查系统性能瓶颈，监控极端场景下的用户体验，避免 “平均响应时间正常，但部分请求超时” 的问题。

------

## 二、指标间核心关系与优化逻辑

### 1. 核心公式

```
系统吞吐量（QPS/TPS） = 并发数 / 平均响应时间
```

- 并发数不变时，**响应时间越短，吞吐量越高**
- 吞吐量不变时，**响应时间越短，可承载的并发数越高**

### 2. 优化优先级

1. **先保障响应时间**：优先将核心接口 RT 控制在用户可接受范围（如接口 RT < 200ms）
2. **再提升吞吐量**：在 RT 稳定的前提下，优化 TPS/QPS，提升系统承载能力
3. **监控极端值**：同步监控最大响应时间，避免长尾请求影响整体体验
