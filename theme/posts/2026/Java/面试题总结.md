# 🧠 Java & Spring 核心面试笔记（对象创建 + Bean + 锁 + 分布式）

------

# 📦 一、对象创建方式总结

## ✅ 除了 `new` 之外的创建方式

### 1. 反射

```java
Class<?> clazz = Class.forName("User");
User user = (User) clazz.getDeclaredConstructor().newInstance();
```

👉 动态创建（框架核心）

------

### 2. clone

```java
User u2 = (User) u1.clone();
```

👉 对象复制（浅拷贝 / 深拷贝）

------

### 3. 反序列化

```java
User user = (User) ois.readObject();
```

👉 IO恢复对象

------

### 4. 工厂方法

```java
User user = UserFactory.create();
```

👉 解耦创建逻辑

------

### 5. 单例模式

```java
User user = Singleton.getInstance();
```

------

### 6. 依赖注入（Spring）

```java
@Autowired
User user;
```

------

## 🎯 总结

👉 对象创建三大类：

- 显式创建：new / 反射
- 复制创建：clone / 反序列化
- 间接创建：工厂 / 单例 / DI

# 🌱 二、依赖注入 & Bean 作用域

## ✅ DI 创建的对象是单例吗？

👉 **不一定，取决于作用域（Scope）**

在 Spring Framework 中：

| 作用域            | 是否单例 |
| ----------------- | -------- |
| singleton（默认） | ✅        |
| prototype         | ❌        |

------

## 📊 singleton vs prototype

| 维度     | singleton    | prototype    |
| -------- | ------------ | ------------ |
| 实例数量 | 一个         | 多个         |
| 生命周期 | Spring全管理 | 只创建不销毁 |
| 是否共享 | 是           | 否           |
| 线程安全 | 不一定       | 相对安全     |

------

## ⚠️ 重点问题

### ❗ prototype 注入 singleton

👉 只会创建一次 ❗

原因：

- 注入发生在 singleton 初始化时

------

## ✅ 解决方案

```java
@Autowired
ObjectFactory<B> factory;

B b = factory.getObject();
```

------

# 🔄 三、Bean 共享 & 设计原则

## ✅ 多个类依赖同一个 Bean

👉 默认 singleton 即可：

```text
A ─┐
   ├──> 同一个 Bean
B ─┘
```

------

## ⚠️ 风险：有状态对象

```java
private int count;
```

👉 多线程共享 → 不安全

------

## ✅ 最佳实践

👉 **无状态设计（推荐）**

------

# 🔁 四、循环依赖（重点）

## ✅ 什么是循环依赖？

```text
A → B → A
```

------

## ✅ Spring 如何解决？

👉 **三级缓存机制**

------

## 🧱 三级缓存

| 缓存 | 作用                  |
| ---- | --------------------- |
| 一级 | 完整 Bean             |
| 二级 | 半成品 Bean           |
| 三级 | Bean 工厂（生成代理） |

------

## 🔄 核心流程

```text
A 创建 → 放入三级缓存
B 需要 A → 从三级拿 → 提前暴露
```

------

## ❗ 为什么需要三级？

👉 为了支持 **AOP 代理对象**

------

# ❌ 五、无法解决的循环依赖

## ❌ 1. 构造器循环依赖

```java
A(B b)
B(A a)
```

👉 原因：

- 必须拿完整对象
- 无法提前暴露

------

## ❌ 2. prototype 循环依赖

👉 原因：

- 不走缓存
- 每次新建 → 无限递归

------

## 🎯 总结

👉 能解决：

```text
singleton + setter
```

👉 不能解决：

```text
构造器 / prototype
```

# 🔐 六、分布式锁（Redis）

## ❌ 错误实现（你的代码问题）

### 1. 递归自旋

```java
return SeachSQL2();
```

👉 ❗ 风险：

- 栈溢出（StackOverflowError）

------

### 2. 没有过期时间

```java
setIfAbsent("lock","1");
```

👉 ❗ 死锁风险

------

### 3. 删除锁不安全

```java
delete("lock");
```

👉 ❗ 误删别人锁

------

## ✅ 正确实现要点

### ✔ 1. 设置过期时间

```java
setIfAbsent(key, value, 10, TimeUnit.SECONDS);
```

------

### ✔ 2. 使用唯一标识

```java
UUID
```

------

### ✔ 3. 循环代替递归

------

## 🚀 推荐方案

👉 使用 Redisson

优点：

- 自动续期
- 可重入
- 防误删
- 高并发安全

# 🔒 七、synchronized 锁升级

## ✅ 锁状态

👉 实际4种：

1. 无锁
2. 偏向锁
3. 轻量级锁
4. 重量级锁

------

## 🔄 升级流程

```text
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

👉 ❗ 不可降级

------

## 📊 对比

| 锁       | 特点    |
| -------- | ------- |
| 偏向锁   | 无竞争  |
| 轻量级锁 | CAS自旋 |
| 重量级锁 | 阻塞    |

------

## 🎯 核心思想

👉 **从无竞争 → 低竞争 → 高竞争逐步升级优化性能**

# 🏁 九、终极总结（建议背）

👉 **标准答案整合版：**

> 在 Spring 中，对象通常由容器通过依赖注入创建，默认是 singleton 作用域，保证全局唯一实例。多个 Bean 共享同一个对象时需要注意线程安全问题，推荐设计为无状态。
>
> 对于多例 prototype，需要注意其注入到 singleton 时只会创建一次，可以通过 ObjectFactory 等方式解决。
>
> 在复杂场景下，Spring 通过三级缓存解决 singleton 的循环依赖，但无法解决构造器和 prototype 的循环依赖。
>
> 在分布式环境中，需要使用 Redis 或 Redisson 实现分布式锁，避免并发问题。底层 JVM 还通过 synchronized 锁升级机制优化性能。
