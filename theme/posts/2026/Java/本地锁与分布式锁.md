# 🔒 本地锁与分布式锁使用指南

------

## 一、核心原则

> **单机用本地锁，分布式用分布式锁；能不用锁就不用锁，锁的粒度要尽量小。**

------

## 二、锁的类型对比

| 锁类型         | 性能 | 适用场景             | 实现复杂度 |
| -------------- | ---- | -------------------- | ---------- |
| **无锁**       | ⚡⚡⚡  | 纯读操作、不可变数据 | 🟢 最简单   |
| **本地读写锁** | ⚡⚡   | 单机、读多写少       | 🟢 简单     |
| **本地重入锁** | ⚡⚡   | 单机、需要互斥       | 🟢 简单     |
| **分布式锁**   | ⚡    | 分布式、强一致性     | 🟡 复杂     |

------

## 三、本地锁的正确用法

### 3.1 读写锁（ReentrantReadWriteLock）

```java
private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
private final Lock readLock = rwLock.readLock();
private final Lock writeLock = rwLock.writeLock();

// 读操作 - 多线程并发
public String readData() {
    readLock.lock();
    try {
        return cache.get("key");
    } finally {
        readLock.unlock();
    }
}

// 写操作 - 互斥执行
public void writeData(String value) {
    writeLock.lock();
    try {
        cache.put("key", value);
    } finally {
        writeLock.unlock();
    }
}
```

**适用场景**：本地缓存、配置管理、读多写少的业务

------

### 3.2 重入锁（ReentrantLock）

```java
private final ReentrantLock lock = new ReentrantLock();

public String getData() {
    lock.lock();
    try {
        // 临界区代码
        return queryDatabase();
    } finally {
        lock.unlock();  // 必须在 finally 中释放
    }
}
```

**适用场景**：单机应用中的互斥操作

------

### 3.3 细粒度锁（避免 synchronized(this)）

```java
// ❌ 错误：锁住整个对象
synchronized (this) {
    // ...
}

// ✅ 正确：只锁关键代码
private final Object lock = new Object();

synchronized (lock) {
    // 只保护需要同步的代码
}
```

------

## 四、分布式锁的使用场景

### 4.1 必须使用分布式锁的场景

| 场景               | 原因                           |
| ------------------ | ------------------------------ |
| **秒杀/抢购**      | 多节点并发扣库存，需要强一致性 |
| **金融交易**       | 资金转账、账户扣款，不允许超卖 |
| **分布式任务调度** | 定时任务只在一个节点执行       |
| **分布式ID生成**   | 雪花算法的 workerId 分配       |

### 4.2 分布式锁实现（Redis）

```java
public boolean executeWithLock(String lockKey, Runnable task) {
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);
    
    if (Boolean.TRUE.equals(locked)) {
        try {
            task.run();
            return true;
        } finally {
            redisTemplate.delete(lockKey);
        }
    }
    return false;
}
```

------

## 五、锁的选择决策树

```
是否需要跨节点同步？
├─ 否 → 用本地锁
│   ├─ 读多写少？ → ReentrantReadWriteLock
│   ├─ 需要互斥？ → ReentrantLock
│   └─ 不需要同步？ → 无锁（ConcurrentHashMap）
└─ 是 → 用分布式锁
    ├─ 强一致性要求？ → Redis/ZK 锁
    └─ 最终一致性？ → 乐观锁、版本号
```

------

## 六、常见误区

| 误区                        | 正确做法                               |
| --------------------------- | -------------------------------------- |
| **滥用 synchronized(this)** | 使用细粒度锁对象                       |
| **分布式场景用本地锁**      | 改用 Redis/ZK 锁                       |
| **锁的范围过大**            | 只锁临界区代码                         |
| **忘记释放锁**              | 使用 try-finally 或 try-with-resources |
| **死锁**                    | 避免嵌套锁、设置超时时间               |

------

## 七、性能对比

### 7.1 响应时间

| 操作                    | 耗时       |
| ----------------------- | ---------- |
| 本地内存读取            | ~100 纳秒  |
| 本地锁（ReentrantLock） | ~1 微秒    |
| Redis 网络通信          | ~1 毫秒    |
| 分布式锁（Redis）       | ~5-10 毫秒 |

### 7.2 并发能力

| 锁类型       | 理论并发数（单机） |
| ------------ | ------------------ |
| 无锁         | 10万+/秒           |
| 读写锁（读） | 5万+/秒            |
| 重入锁       | 1万+/秒            |
| 分布式锁     | 1000+/秒           |

------

## 八、最佳实践

### ✅ 推荐做法

1. **优先考虑无锁方案**：使用 `ConcurrentHashMap`、`AtomicInteger` 等并发工具类
2. **读多写少用读写锁**：提高并发性能
3. **锁的粒度要小**：只锁临界区，不要锁整个方法
4. **必须释放锁**：使用 try-finally 确保锁被释放
5. **分布式场景用分布式锁**：不要用本地锁替代

### ❌ 避免做法

1. **不要滥用 synchronized(this)**：锁粒度太大
2. **不要在分布式场景用本地锁**：无法保证一致性
3. **不要忘记释放锁**：会导致死锁
4. **不要嵌套锁**：容易导致死锁
5. **不要长时间持有锁**：影响并发性能

## 九、代码示例对比

### 9.1 缓存查询（无锁）

```java
// ✅ 最佳：使用 ConcurrentHashMap，无锁
private final Map<String, String> cache = new ConcurrentHashMap<>();

public String get(String key) {
    return cache.get(key);  // 无锁并发读取
}

public void put(String key, String value) {
    cache.put(key, value);  // 无锁并发写入
}
```

### 9.2 缓存查询（读写锁）

```java
// ✅ 适用：需要复杂逻辑的缓存
private final Map<String, String> cache = new HashMap<>();
private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();

public String get(String key) {
    lock.readLock().lock();
    try {
        return cache.get(key);
    } finally {
        lock.readLock().unlock();
    }
}

public void put(String key, String value) {
    lock.writeLock().lock();
    try {
        cache.put(key, value);
    } finally {
        lock.writeLock().unlock();
    }
}
```

### 9.3 缓存查询（分布式锁）

```java
// ✅ 适用：分布式环境下的缓存击穿防护
@Autowired
private RedisTemplate<String, String> redisTemplate;

public String get(String key) {
    String value = redisTemplate.opsForValue().get(key);
    if (value != null) {
        return value;
    }
    
    // 获取分布式锁
    String lockKey = "lock:" + key;
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);
    
    if (Boolean.TRUE.equals(locked)) {
        try {
            // 双重检查
            value = redisTemplate.opsForValue().get(key);
            if (value != null) {
                return value;
            }
            
            // 查询数据库并写入缓存
            value = queryDatabase(key);
            redisTemplate.opsForValue().set(key, value, 1, TimeUnit.HOURS);
            return value;
        } finally {
            redisTemplate.delete(lockKey);
        }
    }
    
    // 未获取到锁，等待或返回默认值
    return "默认值";
}
```

## 十、总结

### 📌 核心要点

1. **本地锁性能远高于分布式锁**：优先使用本地锁
2. **读多写少用读写锁**：提高并发性能
3. **锁的粒度要尽量小**：只锁临界区代码
4. **分布式场景才用分布式锁**：不要滥用
5. **能不用锁就不用锁**：优先考虑无锁方案

### 🎯 一句话总结

> **单机应用用本地锁（读写锁 > 重入锁），分布式应用用分布式锁，能不用锁就不用锁。**
